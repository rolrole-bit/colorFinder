# Plan 2: Antigravity IDE 업데이트 오류 해결 - UI 및 복구 실행 계획

## 1. UI 필요 여부
- **필요 없음**: 본 작업은 시스템 경로 내의 파일 복원 작업이며, 백그라운드에서 실행 파일 유실 상태를 복구하는 것이 목적이므로 별도의 그래픽 UI가 필요하지 않다.
- **이유**: 사용자가 이미 오류 대화상자(UI)를 통해 에러를 인지한 상태이며, 에이전트가 터미널 명령(CLI)을 통해 직접 수동 복구하는 것이 가장 빠르고 직관적이기 때문이다.
- **향후 UI 확장 방식**: 만약 향후 유사한 문제가 반복될 경우를 대비해, 자가 진단 및 복구를 수행할 수 있는 트레이 아이콘 형태의 복구 도우미나 런처 내 복구 버튼(HTML/JS 기반)을 `launcher.hta` 등에 추가하는 방식으로 확장할 수 있다.

## 2. 사용자 흐름
1. 사용자가 업데이트 실패 오류 발생 제보 (완료).
2. 에이전트가 복구 계획 수립 및 복구 프로세스 진행.
3. 복구 완료 후 사용자에게 복구 결과 보고 및 IDE 재실행 요청.

## 3. 화면 또는 명령 흐름
- 복구 명령 흐름:
  1. `Get-Process *antigravity*` 실행하여 구동 중인 프로세스 탐색.
  2. `Stop-Process`로 실행 중인 프로세스 종료.
  3. `robocopy` 또는 `Move-Item` 명령어를 활용하여 `_` 디렉토리 내의 파일들을 상위 디렉토리로 이동.
  4. 이동이 잘 되었는지 파일 목록 재확인 (`Test-Path`).

## 4. Core 모듈과의 연동 방식
- 본 복구 작업은 별도의 Core 애플리케이션 코드를 수정하지 않고, OS 레벨의 파일 시스템 조작 및 프로세스 제어 명령으로 처리한다.

## 5. 상태 처리
- **준비 상태**: 복구 대상 경로(`.../Antigravity IDE/_`) 확인 완료.
- **진행 상태**: 프로세스 종료 시도 및 파일 이동 수행.
- **완료 상태**: 파일 이동이 성공적으로 완료되고 원래 경로에 `Antigravity IDE.exe`가 복원됨.

## 6. 에러 표시
- 명령어 실행 중 발생하는 에러(예: Access Denied, File In Use)는 PowerShell 에러 스트림을 통해 에이전트 콘솔에 상세히 출력되며, 이에 맞추어 대응 조치를 취한다.

## 7. 검증 방법
- `Test-Path "C:\Users\j.rhee\AppData\Local\Programs\Antigravity IDE\Antigravity IDE.exe"` 명령 실행 시 `True`가 반환되는지 검증.
- 복구 완료 후 실행 테스트.

## 8. 확장 가능성
- 향후 Antigravity IDE 업데이트 스크립트에 예외 처리 로직(이동 실패 시 재시도, 권한 상승 요청 등)을 추가하여 업데이트 안정성을 원천적으로 개선할 수 있다.
